Performance

Lighthouse 90点台を出すために
— 静的配信が有利な理由

2025.01  |  読了目安 6分

「ヘッドレスCMSに移行するとLighthouseスコアが上がる」という話をよく聞きます。これは大筋では正しいですが、「静的にすれば自動的に90点台になる」は誤りです。この記事では、スコアに影響する要素と静的配信が有利な理由を正確に説明します。

Lighthouseとは何を測っているか

LighthouseはGoogleが提供するWebパフォーマンス計測ツールです。Chrome DevToolsに組み込まれており、URLを指定するだけで実行できます。スコアはPerformance・Accessibility・Best Practices・SEOの4カテゴリで構成されます。

パフォーマンススコアは主にCore Web Vitals（CWV）と呼ばれる指標群から計算されます。

LCP

Largest Contentful Paint

ページ内の最大コンテンツ（画像・テキストブロック）が表示されるまでの時間。2.5秒以内が良好

INP

Interaction to Next Paint

クリック・タップなどのインタラクションに対するレスポンス速度。200ms以内が良好

CLS

Cumulative Layout Shift

ページロード中のレイアウトのズレ量。0.1以下が良好。画像のサイズ未指定が主な原因

2024年以降、FID（First Input Delay）はINPに置き換えられています。古い記事を参照する際は注意が必要です。

静的配信がLCPに有利な理由

LCPは「最大コンテンツが表示されるまでの時間」です。この指標に最も影響するのはTTFB（Time to First Byte：サーバーが最初のデータを返すまでの時間）です。

WordPressはリクエストのたびにPHP + DBを動かしてHTMLを生成するため、TTFBが数百ms〜1秒以上になることがあります。特にレンタルサーバー（共有ホスティング）で、PHP処理に時間がかかる環境では顕著です。

一方、CDNから静的ファイルを配信する構成では、TTFBは数十ms程度になります。エッジサーバーがすでに持っているHTMLファイルをそのまま返すだけだからです。この差がLCPに直結します。

CLSへの影響

CLSは画像やフォントの読み込みによってレイアウトがズレる現象を測ります。これは静的かどうかよりも、実装の丁寧さに依存します。

  • img要素にwidth/heightを明示していない → レイアウトシフトが発生する
  • Webフォントの読み込みで文字が置き換わる → font-display: swapの設定が必要
  • 遅延読み込み（lazy load）の実装が不適切な場合にもシフトが発生する

Next.jsなどのフレームワークはImage Componentによって自動的にサイズ属性を付与する機能を持ちますが、設定を省略すると改善されません。

静的でもスコアが落ちるケース

「ヘッドレスCMS＝高スコア」と単純化することはできません。以下のような実装ではスコアが低くなります。

  • 大量のJavaScript：Reactベースのフレームワークで不必要にクライアントサイドJSを多用すると、TTI（Time to Interactive）が悪化する
  • 最適化していない画像：WebP変換・適切なサイズのリサイズ・圧縮を行わないと、LCPが大きく低下する
  • サードパーティスクリプト：Google Analytics・チャットウィジェット・広告タグなど外部スクリプトはTBT（Total Blocking Time）に影響する
  • 大量のWebフォント：複数のウェイト・複数のフォントファミリーを読み込むと、LCPが遅延する
  • CDNキャッシュの設定ミス：キャッシュが機能していない場合、オリジンサーバーへの毎回リクエストが発生する
逆に言えば、これらを適切に実装すれば、WordPressでも高スコアを出すことは可能です。キャッシュプラグイン・画像最適化プラグイン・CDNを正しく設定したWordPressサイトはLighthouse 90点台を出せます。「静的かどうか」よりも「実装の質」の方が影響が大きい場面もあります。

静的配信が「有利な理由」の本質

静的配信が高スコアを出しやすい本質的な理由は、「最適化のベースラインが高い」ことです。

WordPressでLighthouse高スコアを実現するには、追加のプラグイン・設定・知識が必要です。キャッシュ・画像最適化・CSS/JSの遅延読み込み・クリティカルCSSの抽出など、それぞれ専門的な設定が要ります。

SSG構成では、これらの多くがフレームワークのデフォルトに含まれています。Next.jsのImage Componentは自動的にWebP変換・リサイズ・遅延読み込みを処理します。静的配信は最初から「速い」のではなく、「最適化された状態になりやすい構造」である、という理解が正確です。

「高スコアを出しやすい構成」という表現について

LPなどで「高スコアを出しやすい構成（静的配信）」という表現を使う理由はここにあります。「確実に90点になる」ではなく「最適化のハードルが低い」という意味です。実際のスコアは実装の質・画像・外部スクリプトの量によって変わります。

Lighthouseのスコアはラボデータ（測定時の環境によって変動する）です。実際のユーザー体験を測るにはCrUX（Chrome User Experience Report）のフィールドデータも合わせて確認することが推奨されます。

まとめ

LCPにはTTFBが直結し、CDN静的配信はTTFBを大幅に改善する。ただし画像の最適化・JSの量・サードパーティスクリプトの影響はどの構成でも存在する。静的配信は「高スコアが保証される」のではなく「最適化のベースラインが高い構成」という理解が正確。
