ヘッドレスCMSとは何か
— WordPressとの構造的な違い

2025.01  |  読了目安 6分

「ヘッドレスCMS」という言葉を聞いたとき、多くの人がまず感じるのは「WordPressとどう違うのか」という疑問です。この記事では、両者の構造的な違いを技術的な観点から整理します。

WordPressの構造：モノリシックCMS

WordPressは「モノリシックCMS（Monolithic CMS）」と呼ばれる構造を持っています。モノリシックとは、コンテンツ管理・テンプレートエンジン・フロントエンド出力がひとつのシステムに統合されている状態を指します。

ブラウザがWordPressのページをリクエストしたとき、サーバー上では次の処理が逐次実行されます。

// WordPressのリクエスト処理フロー（概念） HTTP Request → PHPが起動 → wp-load.php でWordPressを初期化 → DBクエリ（投稿・メタ・オプション取得） → テーマのテンプレートファイルを選択 → HTMLを生成してレスポンス

このフローは毎リクエストのたびに発生します。キャッシュプラグイン（W3 Total Cache、WP Super Cacheなど）は生成済みHTMLをディスクに保存して再利用しますが、キャッシュの設定・管理もまた保守コストになります。

ヘッドレスCMSの構造：デカップルドアーキテクチャ

ヘッドレスCMSは「コンテンツの管理」と「フロントエンドの表示」を分離します。CMSはコンテンツをAPIとして提供するだけで、表示方法には関与しません。

// ヘッドレスCMSのアーキテクチャ（概念） [コンテンツ管理] [ビルドプロセス] [配信] Contentful/Sanity → Next.js/Astro (SSG) → CDN (Vercel/Netlify) ↓ ↓ ↓ APIで提供 ビルド時にHTML生成 静的ファイルを配信

重要なポイントは「ビルド時」という点です。コンテンツを更新したときに一度だけHTMLが生成され、その静的ファイルがCDNに配置されます。ブラウザからのリクエストに対しては、CDNが事前生成済みのHTMLを返すだけです。

SSGとSSRの違い

ヘッドレスCMS構成では、フロントエンドのビルド方式としてSSG（Static Site Generation）とSSR（Server-Side Rendering）の選択があります。

  • SSG：ビルド時に全ページのHTMLを生成。CDNから静的ファイルを配信。リアルタイム性は低いがパフォーマンスが高い
  • SSR：リクエスト時にサーバーでHTMLを生成。リアルタイムデータに対応できるが、サーバー実行環境が必要
  • ISR（Incremental Static Regeneration）：一定時間ごとに静的ファイルを再生成するNext.jsの機能。SSGとSSRの中間的な選択肢

更新頻度が低い小規模サイトにはSSGが最も適しています。ビルド後は純粋な静的ファイルになるため、サーバー実行環境が不要でCDNから直接配信できます。

CDN配信が速い理由

CDN（Content Delivery Network）は世界中にエッジサーバーを持つ配信ネットワークです。ユーザーのリクエストは地理的に最も近いエッジサーバーに届き、事前生成済みのHTMLがそこから返されます。

WordPressの場合、多くの場合は単一サーバー（日本国内1拠点など）から配信されます。CDN配信との差は特に海外ユーザーへのレスポンスタイムに顕著に出ます。国内ユーザー向けであっても、CDNエッジからの配信はオリジンサーバーへのリクエストを大幅に削減します。

コスト構造の違い

WordPressはPHP実行環境・MySQLデータベース・十分なメモリを持つサーバーが必要なため、月3,000〜10,000円程度のVPS・共有ホスティングが一般的なコストです。

SSGによる静的サイトは、HTMLファイルの配信のみでよいため、Vercel・Netlify・Cloudflare Pagesの無料枠で運用できるケースが多くあります。トラフィックが増加すると課金が発生しますが、小規模サイトの通常トラフィックであれば無料〜月1,000円程度で収まることが多いです。

ただし、CMSのコストは別途発生します。Contentfulの無料枠（コンテンツ数に上限あり）やSanityの無料枠（APIリクエスト数に上限あり）の範囲内かどうか確認が必要です。コンテンツ量が多い場合は有料プランが必要になります。

小規模サイトに向いている理由

ヘッドレスCMSへの移行がすべてのサイトに適しているわけではありません。リアルタイムな会員機能・複雑な検索・大量の動的コンテンツがある場合はSSRやWordPressのままの方が適切なケースもあります。

一方、以下のような特性を持つサイトでは、SSG構成が合理的です。

  • ページ数が少ない（20ページ以下）
  • コンテンツの更新頻度が低い
  • リアルタイム性が不要（ランキング・在庫表示などがない）
  • 会員機能・決済が不要

日本には会社案内・士業サイト・小規模店舗のサイトなど、WordPressで構築されているが実態上はほぼ静的なサイトが大量に存在します。これらはSSGに移行しても機能的な差はほとんどなく、パフォーマンス・セキュリティ・保守コストの面で改善が見込めます。

まとめ

WordPressはリクエストのたびにPHP+DBが動作するモノリシック構造。ヘッドレスCMSはビルド時にHTMLを生成し、CDNから静的ファイルを配信するデカップルド構造。更新頻度が低い小規模サイトでは、後者の方が構造的にシンプルでコストも低い。
