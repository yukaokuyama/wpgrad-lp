Security

WordPressが攻撃されやすい理由と、
静的サイトで変わること

2025.01  |  読了目安 7分

「WordPressはセキュリティリスクが高い」という話はよく聞きます。しかし「なぜ高いのか」「静的サイトにするとどこが変わるのか」を正確に理解している人は意外と少ない。この記事では攻撃経路を一つひとつ整理します。

WordPressが攻撃対象になりやすい構造的な理由

WordPressがセキュリティ的に難しい理由は、シェアの高さと構造の複雑さの両方にあります。

世界のWebサイトの約40%がWordPressで動いています。これは攻撃者にとって、一つの脆弱性を見つければ膨大な数のサイトに適用できることを意味します。攻撃の多くは自動化されたボットによる大規模スキャンで、特定のサイトを狙うのではなく「インターネット上のWordPressサイト全体」を対象にしています。

主な攻撃経路

① wp-login.php への総当たり攻撃（ブルートフォース）

WordPressのログインURL（/wp-login.php）は全インストールで共通です。ボットは大量のIDとパスワードの組み合わせを試し続けます。

// 典型的なログインスキャン（概念） POST /wp-login.php user: admin pass: password123 → 失敗 POST /wp-login.php user: admin pass: admin123 → 失敗 POST /wp-login.php ...（数千〜数万回繰り返す）

対策としてログインURLの変更・2FAの導入・IP制限などがありますが、これらはプラグインや設定変更が必要で、維持管理のコストが発生します。

② プラグインの脆弱性

WordPressの脆弱性の大半はコアではなくプラグインに存在します。WPScanのデータベースには数万件のプラグイン脆弱性が登録されています。代表的な脆弱性の種類は以下のとおりです。

  • SQLインジェクション（SQLi）：入力値を適切にエスケープせずDBクエリに渡すことで、不正なSQLを実行される
  • クロスサイトスクリプティング（XSS）：管理画面や投稿に不正なJavaScriptを注入される
  • リモートコード実行（RCE）：任意のコードをサーバー上で実行される。最も深刻なクラス
  • 権限昇格：認証済みの低権限ユーザーが管理者権限を取得する
  • ファイルインクルージョン：サーバー上の任意ファイルを読み込まれる

問題は、これらの脆弱性が「プラグインのアップデート」によって修正されますが、更新しないまま放置されたサイトが多数存在することです。更新頻度が低い小規模サイトほど、このリスクが高くなります。

③ データベースへの不正アクセス

WordPressはコンテンツをMySQLデータベースに格納しています。SQLiが成立した場合、攻撃者はユーザーのパスワードハッシュ・メールアドレス・サイト設定などを取得できます。

また、wp-config.php にはデータベースの接続情報（ホスト・ユーザー名・パスワード）が平文で記載されています。このファイルへのアクセス制限が不十分な場合、直接読み取られるリスクもあります。

④ xmlrpc.php の悪用

xmlrpc.php はWordPressのリモートAPI機能を提供するファイルです。現在はREST APIに置き換えられましたが、後方互換性のために残っています。この機能を悪用した増幅攻撃（1リクエストで複数のログイン試行が可能）やDDoS攻撃の踏み台としての利用が知られています。

静的サイトへの移行で変わること

SSGによる静的サイト構成では、上記の攻撃経路の多くが構造的に存在しなくなります。

攻撃経路 WordPress 静的サイト（SSG）
wp-login.php への攻撃 存在する 存在しない（管理画面は別ドメインのCMS）
プラグイン脆弱性 常に管理が必要 WordPressプラグインは存在しない
データベースへの攻撃 存在する（SQLi等） 配信サーバーにDBなし
xmlrpc.php 悪用 存在する 存在しない
PHPコード実行 PHPが動作している PHPが存在しない（静的ファイル配信）

静的サイトでも残るリスク

正確に理解するために、移行しても残るリスクも明示します。

  • フォームの実装：お問い合わせフォームをFormspreeやfunctionなどのサービスで実装する場合、そのサービス自体のセキュリティに依存する。スパム・フィッシング対策が必要
  • CMS管理画面：ContentfulやSanityなどのCMS管理画面は引き続きログインが必要。2FA設定・強力なパスワード管理は必須
  • CDN・ホスティングの設定ミス：Vercel・Netlifyの設定を誤ると、意図しないファイルが公開されるリスクがある
  • 依存パッケージの脆弱性：Next.js・Astroなどのフロントエンドフレームワークとnpmパッケージも定期的な更新が必要
  • GitHubリポジトリの公開設定：ソースコードをパブリックリポジトリで管理する場合、API KeyやTokenをコードに含めてしまうリスクがある
「静的サイトにすればセキュリティリスクがゼロになる」は誤りです。正確には「WordPress特有の攻撃経路が構造的に減る」です。残存するリスクへの対応は引き続き必要です。

保守の観点から見たリスク管理コスト

WordPressのセキュリティ対策で特にコストがかかるのは「継続的な更新管理」です。コアのアップデート・プラグインのアップデート・PHPバージョンの追従・バックアップの確認。これらは月次〜週次で発生する作業です。

一方、SSG構成ではWordPressコア・プラグインの更新は不要になります。フロントエンドフレームワーク（Next.jsなど）の更新は必要ですが、頻度・緊急度はWordPressのプラグイン更新より低い傾向があります。

まとめ

WordPressは世界シェア40%ゆえに大規模スキャンの標的になりやすく、ログイン画面・プラグイン・DB・xmlrpc.phpが主な攻撃経路。SSGへの移行でこれらの経路は構造的に減る。ただしフォーム・CMS管理画面・CDN設定ミスなど残存リスクも存在するため、「ゼロリスク」という表現は正確ではない。
